# tinyOS Roadmap — Toy to Real Without Bloat

Date: 2026-07-19
Status: agreed direction from the 2026-07-19 architecture review

Companion spec: `2026-07-19-terminal-and-crates-design.md` (crate
organization + terminal stack). Its milestones M1–M4 interleave with the
phases below; cross-references are noted per phase.

## Where we are

Strengths already above hobby-OS grade — build on, don't rework:

- Real EL0/EL1 isolation: per-process TTBR1 + ASID-tagged flush-free
  switches; kernel pages EL1-only; loader-enforced W^X; user pointers
  validated against recorded mappings before every kernel copy.
- The capability ABI (13 syscalls: handles + rights, channels, memobjs,
  one unified wait), proven end-to-end by solitaire at EL0 with
  zero-copy shared-memory graphics and timer preemption.
- tinyfs: shadow-paging CoW whose crash test replays every write-log cut
  point and proves each prefix mounts as exactly old-or-new.
- Concurrency discipline: deferred-handoff context switch, IRQ handlers
  that only set flags, tickless idle.

The gaps are not where hobby OSes usually have them. They concentrate in
**memory-management foundations**, the **single-app ceiling**, and **how
much of the OS still runs at EL1** (compositor, full shell, five apps).

## Standing rule

**All new kernel APIs are handle/channel-shaped.** Files, process
control, terminals — protocols over channels, even when the first server
is in-kernel. Never bespoke syscalls per service. This keeps "where does
this code run" a deployment decision, and it is what makes the Phase 4
eviction and the later userspace-`fsd` re-host cheap.

## Phase 1 — Memory foundations (unblocks everything)

The least glamorous and highest-leverage work. Today "frames" are 4 KiB
heap blocks from one `linked_list_allocator` over only the single
largest UEFI region (`mem/frames.rs`, `mem/mod.rs`): kernel heap, page
tables, stacks, and all user memory share one pool with no quotas, and
mapped VA is never reclaimed (`alloc_va` is a bump with no unmap).
Consequence: any app can `memobj_create` its way to kernel-heap
exhaustion, and long-lived processes leak their own VA window.

1. **Real frame allocator.** Bitmap over *all* conventional UEFI
   regions (no buddy system needed). Keep the existing phys-addr-shaped
   API so callers don't change.
2. **Per-process memory quotas.** Charge MemObj frames, page-table
   frames, and stacks to the owning process; cap with a sane default.
   Turns heap-exhaustion DoS into a per-process failure.
3. **`memobj_unmap` (reserved syscall #15) + VA reclaim.** Replace the
   bump `alloc_va` with a free-list that can reuse ranges; drop mapping
   records on unmap. Note: this invalidates the current user-pointer
   TOCTOU argument ("no unmap exists"), so `copy_in`/`copy_out`
   validation must hold the aspace lock across the copy, or pin the
   mapping for the syscall's duration — decide when implementing.
4. **Guard pages on kernel stacks.** 64 KiB stacks currently sit in the
   heap; overflow silently corrupts adjacent allocations.
5. **Window-surface trust audit.** Verify the compositor blits strictly
   from the kernel-side MemObj object a handle refers to — never from
   any address or size the app claims in a message
   (`ui/shell/extern_app.rs`). Fix if not. (The terminal emulator
   inherits this rule; the terminal spec states it explicitly.)

## Phase 2 — Multi-process

- Implement `process_spawn` (reserved syscall #13) with explicit handle
  grants, per the app-api spec's capability model.
- Delete the one-running-app limit in the terminal (`term/mod.rs`); the
  shell pumps N extern-app channels; the compositor already handles
  multiple windows.
- The fixed app link base is *not* a blocker — every process has its own
  TTBR1, so identical VAs across processes are free (per the app-api
  spec).
- `thread_spawn` (reserved #14) can wait until something needs it.

Terminal-spec M2 (console v1 line world, stdin) lands well here: spawn +
stdin together make the terminal a real multi-program host.

## Phase 3 — Files over channels

A filesystem protocol (`crates/abi/fs.rs`): open/read/write/readdir/
stat/rename as messages; per-file and per-directory capability handles;
offset-based I/O in the protocol from day one so a cache can slot in
later without ABI change. Served initially by a kernel-side endpoint,
granted at bootstrap like console/shell. The syscall table does not
grow.

Also fixes iteration friction: a writable `/apps` plus an `install` path
replaces re-baking `disk.img` at mkfs time.

## Phase 4 — Eviction

Move the in-kernel desktop across the syscall boundary, one piece at a
time, each port dogfooding the SDK:

1. notes, clock, monitor, editor → SDK apps (the editor port doubles as
   terminal-spec M3's proof app).
2. The shell: built-ins re-based onto the fs protocol (Phase 3) and a
   narrow, capability-gated **process-control protocol** (`ps`/`kill`/
   `memstat` as messages any sufficiently-privileged app could speak —
   not a fat "kernel control" channel that is syscalls by another name).
3. The terminal emulator itself becomes an SDK app (terminal spec,
   "future").
4. Delete the in-kernel `App` trait fast-path.

End state: kernel = compositor/WM, drivers, scheduler, MM, object layer.

## Second wave (after Phases 1–4)

- **Userspace `fsd`.** Re-host the Phase-3 fs service as a userspace
  process (needs spawn + a block-device capability grant). Clients see
  no change. This is the designated proof that the channel boundary is
  real.
- Per-object wait queues (replace the three global thundering-herd
  queues and the 100 ms `wait_many` poll cap).
- Interrupt-driven virtio-blk completion, so slow I/O stops wedging the
  UI thread.

## Deliberately not doing (bloat list)

- **No POSIX layer, no fork, no signals, no errno** — in the kernel,
  ever (app-api pillar; a userspace compat layer remains possible).
- **No dynamic linking / PIE / ASLR.** Fixed-base static ELF is fine
  given per-process address spaces.
- **No demand paging, CoW, or swap** until something needs them.
- **No x86_64 ring-3 parity chase.** Keep it compiling; revisit after
  Phase 4. One real target (aarch64/HVF) until the design stabilizes.
- **No page cache.** tinyfs whole-file I/O is slow but correct and fine
  at QEMU scale; the fs protocol's offset-based ops leave the door open.
- **No userspace drivers.** They need interrupt-delivery objects,
  DMA-safe grants, and IOMMU containment we don't have — much new kernel
  machinery for ~800 lines of QEMU-only drivers. Virtio stays in-kernel.
- **The compositor stays in-kernel** indefinitely; the window protocol
  already isolates apps from that choice.
- **No real-hardware hardening** (DMA cache maintenance, ACPI/DT
  discovery) unless leaving QEMU becomes a goal; document the
  assumption instead.

## Cheap wins (any time)

- README refresh: it still says "no processes (yet)".
- Bounds-check the 8-slot `INPUT_ISR_ADDRS` claim path.

## Sequencing at a glance

| Step | Delivers |
|---|---|
| abi M1 + Phase 1 | Safe foundations: frames, quotas, unmap, guards, trust audit |
| Phase 2 + terminal M2 | Many processes, real stdin — a multitasking desktop |
| Phase 3 | Apps get files; app install without re-baking disk.img |
| Phase 4 + terminal M3/M4 | Desktop moves to userspace; kernel shrinks to its core |
| Second wave | fsd re-host, per-object waits, async blk |
