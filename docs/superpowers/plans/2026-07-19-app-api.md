# App API — Implementation Notes

Date: 2026-07-19
Spec: [2026-07-18-app-api-design.md](../specs/2026-07-18-app-api-design.md)

Milestone 1 delivers a capability-based userspace on aarch64: real EL0
processes loaded from the ESP, a ~13-syscall ABI, message-passing IPC, and a
`tinyos-app` SDK. x86_64 keeps building; its ring-3 backend is a follow-up
behind the same ABI.

## Verification

No hosted test harness; verification is `make run` plus in-kernel test
commands (kernel-side unit tests) and fw_cfg boot hooks for headless CI-style
checks:

- `usertest` / `boottest=user` — hand-assembled EL0 code logs via SVC + exits.
- `usertest spin` / `boottest=spin` — proves EL0 timer preemption + `kill`.
- `objtest` / `boottest=obj` — 9 object-model assertions.
- `boottest=sys` — clock + `wait_many` timed sleep + exit from EL0.
- `ls` / `boottest=fs` — tinyfs mounts `disk.img` and lists `/apps`.
- `run hello a b c` / `boottest=run` — loads hello off `/apps` (tinyfs),
  streams its console output (argv echo), reports `exited (code 3)`. Verified
  reliable across repeated headless runs.

Originally milestone 1 shipped its own read-only FAT reader + polled virtio-blk
to load apps off the ESP. When tinyfs (a native CoW filesystem) landed on
main, that redundant driver and FAT reader were removed and app-loading was
rewired onto the tinyfs facade (`fs::read`/`fs::list`). App binaries are now
baked into the tinyfs `disk.img` at build time (`mkfs-tinyfs --populate`);
`make cleandisk` refreshes them after an app rebuild.

### HVF note

The `boottest=run` headless harness drains the app's console from a busy
kernel thread. A busy-yielding vCPU pegs a host core under HVF and starves
the emulated APs running the app, so the harness paces itself with a periodic
serial write. The real terminal drains from a frame-driven `pump()` that
blocks between frames and never busy-loops, so it is not affected.

## Known follow-ups

- **Window compositor host**: the shell needs an `App`-trait host that, on a
  window-protocol `OPEN`, maps the app's shared surface and blits it, and
  forwards input events. The client (`apps/sdk/src/window.rs`) and demo
  (`apps/pixels`) are ready.
- **Thread/aspace reclamation**: an exiting user thread is retired via
  `Handoff::Drop`, but the process's address space is currently only fully
  reclaimed when its last `Arc` drops; wire deterministic teardown.
- **`sleep_us` from kernel threads**: the TIMER-waitq wake path did not
  reliably wake a lone polling kernel thread in testing; the object/`INPUT`
  wait paths are unaffected. Worth a focused look.
- **x86_64 ring 3**: GDT/TSS + `SYSCALL`/`SYSRET` behind the same ABI.
- **Real frame allocator**: replace the identity-map heap façade.
